# CX — Architecture

> See [`docs/overview.md`](../overview.md) for project context, problem statement, and success criteria.

---

## Design Principles

### 1. The Binary Is the Agent's Tool, Not the Developer's

Developers never call `cx` commands directly for spec work. They talk to their coding agent, and the agent — guided by skills — calls `cx` commands. The only direct developer interaction with `cx` is:

- `cx init` — one-time project setup
- `cx dashboard` — TUI for visibility into the system

Everything else flows through: **Developer → Agent → Skill → cx command → docs/**

### 2. No Config File

The project has exactly two resources: the `cx` binary and the `docs/` directory. There is no `cx.yaml`, no `.cxrc`, no config file of any kind. The binary infers everything it needs from:

- **docs/** content — project info, tech stack, conventions, specs
- **Git config** — author identity (`git config user.name`)
- **Auto-detection** — which agent directories exist, project type from manifest files
- **Conventions** — standardized docs/ structure means predictable behavior

This eliminates config drift, reduces surface area, and means docs/ truly is the single source of truth.

### 3. Deterministic Commands, Non-Deterministic Agents

Every `cx` command produces the same output given the same input, regardless of which agent calls it. The agent handles understanding intent and choosing commands. The binary handles executing them reliably.

### 4. Append-Only Memory with Git as Sync

No CRDTs, no conflict resolution strategies. Observations and decisions are append-only markdown files committed to git. Team sync is just git — `git push` shares your memories, `git pull` gets the team's. A `post-merge` git hook triggers FTS5 index rebuild and conflict detection when new memory files arrive. The index cache (`.cx/.index.db`) is never the source of truth. All entity relationships use a single `deprecates` field — no separate `supersedes` mechanism.

Personal notes (local-only) live in `~/.cx/memory.db` (SQLite). They support upsert and deletion via `topic_key`.

### 5. Git History Is the Spec Version System

No separate versioning for specs. Git provides the full history of docs/ evolution. The `archive/` directory provides human-readable audit trail of completed changes.

### 6. Atomic Writes

All `cx` write operations use the temp file + atomic rename pattern: write to a `.tmp` file in the same directory, then `os.Rename()` to the final path. This prevents partial files if the process is interrupted. Combined with the append-only memory model, this means the system is always in a consistent state — either the file exists completely or it doesn't exist at all. If the index cache becomes stale or corrupted, `cx index rebuild` regenerates it from the source files.

## System Architecture

```
Developer ──→ Coding Agent ──→ reads skills ──→ calls cx ──→ reads/writes docs/
    │                │                                │
    │                │                                └──→ reads/writes .cx/
    │                │
    │                └──→ MCP servers (Linear, etc.) ──→ external APIs
    │
    └──→ cx dashboard (TUI) ──→ reads docs/ + .cx/ (read-only view)
```

## Commands

### Developer-Facing (direct use)

| Command | Purpose |
|---------|---------|
| `cx init` | Bootstrap agent directories + skills. Auto-detects agents. Creates `.cx/`. Registers project globally |
| `cx upgrade` | Update binary via Homebrew, then `cx sync` all registered projects |
| `cx projects` | List all cx-initialized projects registered in `~/.cx/projects.json` |
| `cx projects remove <path>` | Remove a project from the global registry |
| `cx dashboard` | TUI: projects, changes, memory, specs, team sync |

### Agent-Facing (called via skills)

**Context priming** (called by primer subagent — see [context-priming spec](../specs/context-priming/spec.md)):

| Command | Purpose |
|---------|---------|
| `cx context --mode <mode>` | Step 1: return the context map for this session mode |
| `cx context --load <resource> [name]` | Step 2: load full content of a specific spec, change, or doc |

**Memory** (called by main agent during work):

| Command | Purpose |
|---------|---------|
| `cx memory save --title "..." --type ... --content "..."` | Save observation |
| `cx memory decide --title "..." ...` | Record a technical decision |
| `cx memory session --goal "..." ...` | Save session summary |
| `cx memory note --title "..." ...` | Save personal note (local only) |

**Search & Index** (called by agents for discovery — see [search spec](../specs/search/spec.md)):

| Command | Purpose |
|---------|---------|
| `cx search "query"` | Unified FTS5 search across all of docs/ |
| `cx search "query" --memory` | Scope to docs/memories/ only |
| `cx search "query" --specs` | Scope to docs/specs/ only |
| `cx search "query" --changes` | Scope to docs/changes/ only |
| `cx search "query" --personal` | Include personal notes from ~/.cx/memory.db |
| `cx index rebuild` | Rebuild FTS5 index from all markdown in docs/ |

**Brainstorm & Change lifecycle** (see [brainstorm spec](../specs/brainstorm/spec.md) and [change-lifecycle spec](../specs/change-lifecycle/spec.md)):

| Command | Purpose |
|---------|---------|
| `cx brainstorm <name>` | Create masterfile template in docs/masterfiles/ |
| `cx decompose <name>` | Transform masterfile into a change structure, archive masterfile |
| `cx change new <name>` | Create a change directly (without brainstorm phase) |
| `cx change status` | List active changes with completion state |
| `cx change archive <name>` | Validate, merge deltas → specs, move to archive |

**Conflict resolution** (called by conflict-resolver subagent — see [conflict-resolution spec](../specs/conflict-resolution/spec.md)):

| Command | Purpose |
|---------|---------|
| `cx conflicts detect` | Compare new entities against index, write .cx/conflicts.json |
| `cx conflicts list` | Print current unresolved conflicts |
| `cx conflicts clear` | Delete .cx/conflicts.json without resolving |

**System**:

| Command | Purpose |
|---------|---------|
| `cx sync` | Regenerate agent configs + skill files for the current project |
| `cx doctor` | Validate docs/ structure, memory health, index, git hooks, MCP, skills |
| `cx doctor --fix` | Auto-repair fixable issues (dirs, hooks, index, skills) |
| `cx index rebuild` | Rebuild FTS5 index from docs/ |

### Author Identity

The binary reads the author from git — no environment variable needed:

```bash
git config user.name   # → "angel"
git config user.email  # → "angel@company.com"
```

The `user.name` is used as the author field in memory files and the filename slug. Every developer already has this configured. This means:
- Zero setup for author identity
- Memory file author matches the git commit author
- No `CX_AUTHOR` env var to remember or configure

### MCP Dependencies

CX does **not** integrate with external APIs directly. External services (Linear, Slack, GitHub, etc.) are accessed by the agent through MCP servers. On `cx init`, the binary checks `.mcp.json` for required MCP servers and prompts the user to configure any that are missing.

`cx doctor` also validates MCP dependencies and warns about missing servers.

### Git Hooks

`cx init` installs hooks into `.git/hooks/` to keep the FTS5 index fresh when memory files arrive via git. No remote servers, no custom sync protocol — git is the sync layer.

**`post-merge`** — runs after `git pull`:
```bash
#!/bin/sh
if git diff-tree -r --name-only ORIG_HEAD HEAD | grep -q '^docs/'; then
    cx index rebuild 2>/dev/null || true
    cx conflicts detect 2>/dev/null || true
fi
```

**`post-checkout`** — runs after branch switch:
```bash
#!/bin/sh
# $3 is 1 for branch checkout, 0 for file checkout
if [ "$3" = "1" ]; then
    cx index rebuild 2>/dev/null || true
fi
```

Both are silent, non-blocking (`|| true`), and fast (<100ms). If the hooks aren't installed, correctness isn't affected — the binary also checks mtimes on every command and rebuilds lazily. The hooks just make it proactive.

`cx doctor` warns if hooks are missing. `cx init` asks before overwriting existing hooks.

## docs/ Directory Structure

### No Frontmatter, No Metadata Files

The docs/ directory is pure markdown. The binary parses it by convention, not configuration:

```
docs/overview.md                  → cx reads project name from first H1
docs/architecture/index.md        → cx reads tech stack from "Tech Stack" section
docs/specs/index.md               → cx reads spec tree from the file content
docs/memories/DIRECTION.md        → cx reads memory policy for agent guidance
docs/memories/observations/       → cx reads + writes team observations
docs/memories/decisions/          → cx reads + writes team decisions
docs/memories/sessions/           → cx reads + writes session summaries
docs/masterfiles/                 → cx reads active brainstorm documents
docs/changes/*/                   → cx reads change state from file existence
docs/archive/                     → completed changes (not indexed)
.cx/.index.db                     → local FTS5 cache covering all docs/ (never read as source of truth)
```

### Tech Stack

| Layer | Technology | Purpose |
|-------|------------|---------|
| Language | Go | Single static binary, fast, cross-compilable |
| CLI | cobra | Command structure and flag parsing |
| TUI | Bubble Tea | Dashboard terminal UI |
| Search | SQLite FTS5 | Full-text search over memory entities |
| Local memory | SQLite | Personal notes (upsertable, never synced) |
| Project memory | Markdown + git | Append-only observations/decisions/sessions |
| Task tracking | Linear via MCP | Agent connects through MCP server |
| Distribution | Homebrew, npm wrapper, GitHub Releases | Cross-platform install |

### Spec Structure

```
docs/specs/
├── index.md                       # Tree view + description of each spec area
├── <area>/
│   ├── spec.md                    # WHAT the system does (requirements)
│   └── scenarios.md               # HOW it behaves (Given/When/Then)
└── ...
```

### Change Structure

```
docs/changes/<name>/
├── proposal.md                    # WHY: intent, scope, approach
├── design.md                      # HOW: technical decisions, architecture
├── tasks.md                       # WHAT: Linear issue references + implementation notes
└── specs/                         # DELTA: what changes in existing specs
    └── <area>/
        └── delta.md               # ADDED/MODIFIED/REMOVED sections
```

### Delta Spec Format

```markdown
# Delta: <spec-area>

## ADDED
### Requirement: <name>
<requirement text>

#### Scenario: <name>
- GIVEN ...
- WHEN ...
- THEN ...

## MODIFIED
### Requirement: <name> (from spec.md line N)
<new text>
(Previously: <old text>)

## REMOVED
### Requirement: <name>
(Deprecated — <reason>)
```

## Feature Lifecycle

![Feature Lifecycle](diagrams/03-feature-lifecycle.mermaid)

The full lifecycle from idea to archived spec change:

1. **Brainstorm** — Agent runs `cx brainstorm`, fills masterfile.md, iterates via skill-guided refinement
2. **Decompose** — Agent runs `cx decompose`, binary creates docs/changes/ structure
3. **Specify** — Agent fills proposal.md, design.md, tasks.md; creates Linear issues via MCP
4. **Implement** — Agent writes code, saves observations with `cx memory save`, updates Linear via MCP
5. **Archive** — Agent runs `cx change archive`; binary merges deltas → specs, moves to archive/

## Skills — The Communication Protocol

Skills are the interface between agents and the binary. Every skill follows the same structure:

```markdown
# Skill: cx-<command>

## Description
What this skill does and when to use it.

## Triggers
- Natural language patterns that activate this skill

## Steps
1. Exact cx command to run
2. How to interpret the output
3. What to do next

## Rules
- Constraints and guardrails
```

![Skill Protocol](diagrams/07-skill-protocol.mermaid)

### Context Priming — The Primer Subagent

The most critical skill is context priming. It doesn't run in the main agent — it runs in a **dedicated primer subagent** spawned at session start.

```
Developer writes opening message
    → Main Agent spawns primer subagent (passes developer's message)
        → Primer classifies mode: CONTINUE | BUILD | PLAN
        → Primer calls cx context --mode <mode> (gets the map)
        → Primer evaluates relevance, calls cx context --load for specifics
        → Primer distills ~500-800 tokens of focused context
    → Main Agent receives primed context and begins work
```

The primer's context window is disposable. It can load the full spec index, multiple spec files, all active decisions, recent observations — reason about what's relevant — then output a tight context block. The main agent never sees the noise.

See [context-priming spec](../specs/context-priming/spec.md) for the full design.

### Core Skills

| Skill | Used by | Trigger | Key Commands |
|-------|---------|---------|-------------|
| `cx-prime` | Primer subagent | Session start (always) | `cx context --mode`, `cx context --load` |
| `cx-memory` | Main agent | During/end of work | `cx memory save/decide/session` |
| `cx-brainstorm` | Main agent | New idea | `cx brainstorm`, `cx decompose` |
| `cx-refine` | Main agent | Masterfile iteration | Skill-guided (no binary commands) |
| `cx-change` | Main agent | Structured work | `cx change new/status/archive` |
| `cx-linear` | Main agent | Task tracking | Linear MCP server calls |
| `cx-reverse` | Reverse-eng subagent | Existing codebases | `cx search`, `cx context --load` |
| `cx-doctor` | Main agent | Health checks | `cx doctor`, `cx doctor --fix` |
| `cx-conflict-resolve` | Conflict-resolver subagent | After git pull (conflicts detected) | `cx conflicts detect/list`, `cx memory decide` |

### Skill Generation

`cx init` and `cx sync` generate skills tailored to each agent:

- **Claude Code**: `.claude/skills/cx-*.md` — uses Claude's skill format with `## Steps` and bash blocks
- **Gemini**: `.gemini/skills/cx-*.md` — adapted for Gemini's instruction format
- **Codex**: Embedded in `AGENTS.md` — Codex prefers single-file instructions

The content is identical across agents. Only the format and file location differ.

## Linear Integration (via MCP, not cx)

The binary has zero integration with the Linear API. All Linear communication happens through the agent's MCP server.

| Concern | Owner |
|---------|-------|
| Creating/updating Linear issues | Agent via Linear MCP server |
| Reading Linear issue status | Agent via Linear MCP server |
| Managing tasks.md structure | cx binary |
| Parsing Linear refs from tasks.md | cx binary |

`tasks.md` is a reference document that maps implementation units to Linear issues — not a to-do list with checkboxes.

```markdown
# Tasks: <change-name>

## Linear Issues
- CUB-140: <task description>
- CUB-141: <task description>

## Implementation Notes
<technical notes relevant to implementation>
```

## Implementation Plan

### Phase 1: Core (MVP)
- Go binary skeleton with CLI (cobra), atomic writes (temp file + rename)
- `cx init` — agent selection, skill generation, DIRECTION.md setup, global project registration, auto-update prompt
- `cx sync` — regenerate agent configs from docs/, always overwrite skills
- `cx upgrade` — `brew upgrade cx` + iterate `~/.cx/projects.json` + `cx sync` each project
- `cx projects` / `cx projects remove` — manage global project registry
- `cx change new / status / archive` — change lifecycle with agent-assisted spec merge
- `cx doctor` / `cx doctor --fix` — docs/ validation + auto-repair
- Core skills: cx-change.md, cx-memory.md, cx-doctor.md

### Phase 2: Memory + Search + Context Priming
- Markdown-based project memory (docs/memories/observations/, decisions/, sessions/)
- Unified `deprecates` field for all entity relationships (no separate `supersedes`)
- SQLite FTS5 index cache (.cx/.index.db) covering all of docs/
- `cx index rebuild` — explicit FTS5 rebuild command with mtime-based lazy trigger
- Git hooks: `post-merge` and `post-checkout` trigger `cx index rebuild` + `cx conflicts detect`
- `cx memory save / decide / session` — write memory entities
- `cx search` — unified search with scope flags (--memory, --specs, --changes, --personal)
- `cx context --mode / --load` (two-step context priming)
- `cx conflicts detect / list / clear` — semantic conflict detection
- SQLite personal notes (~/.cx/memory.db) with upsert via topic_key
- Primer subagent skill: cx-prime.md
- Conflict-resolver subagent skill: cx-conflict-resolve.md
- Skills: cx-memory.md

### Phase 3: Brainstorm + Decompose
- `cx brainstorm <name>` — masterfile template generation in docs/masterfiles/
- `cx decompose <name>` — masterfile → docs/ change structure + archive
- Refinement skill (cx-refine.md) — agent-guided masterfile iteration (no binary command)
- Skills: cx-brainstorm.md, cx-refine.md

### Phase 4: Reverse Engineering + MCP Validation
- Reverse-engineering subagent skill (cx-reverse.md) — codebase exploration via agent tools + cx search
- `.mcp.json` validation in `cx init` and `cx doctor`
- Skills: cx-reverse.md, cx-linear.md (MCP-based instructions)

### Phase 5: Dashboard
- Bubble Tea TUI framework
- All five tabs: projects, changes, memory, specs, team

### Phase 6: Distribution
- Homebrew formula (primary distribution — `cx upgrade` depends on this)
- GitHub releases (cross-compiled binaries for Homebrew and direct download)
- npm binary wrapper (optional convenience)
- Install script (`curl | sh`)
